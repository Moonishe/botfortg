# Constitution — TelegramHelper v2.0

Ты работаешь над проектом **TelegramHelper v2.0** и обязан соблюдать эту конституцию. Она имеет **высший приоритет** над всеми остальными инструкциями (rules.md, AGENTS.md, etc.), кроме прямого запроса пользователя.

---

## Authority (Порядок приоритетов)

При конфликте инструкций — соблюдай этот порядок:

1. **Текущий запрос пользователя** — высший приоритет
2. **Живой код и тесты** — код есть истина; что проходит тесты, то правда
3. **AGENTS.md** — проектные инструкции
4. **Глобальные правила** (~/.config/opencode/rules.md)
5. **Память** — DSM + .opencode/memory/
6. **Handoffs** — из предыдущих сессий

---

## Protected Invariants (Железные правила)

Эти правила **нельзя нарушать ни при каких обстоятельствах**:

1. **Асинхронность**: ВСЕ I/O через async/await. Sync-обёртки только через `asyncio.to_thread()`.
2. **Конфигурация**: Только через pydantic-settings из `.env`. **НИКАКИХ секретов в коде**.
3. **База данных**: Миграции через Alembic. FTS5 через `init_db()`. **НИКАКИХ сырых SQL**.
4. **Ошибки**: Логировать ВСЕ исключения. Никаких `bare except: pass`.
5. **Graceful shutdown**: Каждый компонент должен иметь cleanup с таймаутами.
6. **Типы**: Аннотации на ВСЕХ публичных функциях/методах (pyright strict mode).
7. **Lint**: Ruff с полным набором правил (S — security, PT — pytest, TCH — type checking, PIE — performance).

---

## Branch Policy

- **main** — основная ветка, только проверенный код
- **feature/\<name\>** — для новых фич
- **fix/\<description\>** — для исправлений
- Работай в feature/fix ветке. PR в main только после прохождения Zero-Risk Pipeline.

---

## Verification Policy (Перед тем как сказать «готово»)

Прежде чем объявить работу завершённой, ты ОБЯЗАН:

1. Запустить тесты: `pytest tests/ -x -v` (in-memory SQLite)
2. Прочитать все изменённые файлы и проверить diff
3. Убедиться что Zero-Risk Pipeline (D5→R5) завершён с **0 проблем**
4. **Никогда не утверждать верификацию, которую не выполнял**

---

## Escalate When (Когда остановиться и спросить)

Немедленно остановись и запроси разрешение пользователя, если:

- Деструктивное действие без явного разрешения (удаление данных, drop table)
- Изменение провайдеров / аутентификации / `.env` / `opencode.jsonc`
- Удаление или перезапись файлов, которые ты НЕ создавал
- Изменение ORM-моделей (`src/db/models/`) без разрешения
- Изменение Alembic-миграций (`alembic/versions/`) без разрешения
- Изменение `src/config.py` (Settings class) без разрешения
- Zero-Risk Pipeline зациклился (10+ итераций D5→R5)

---

## Zero-Risk Pipeline

После ЛЮБЫХ изменений кода:

1. **D5**: 5 параллельных debugger'ов ищут и исправляют проблемы
2. **R5**: 5 параллельных reviewers проверяют код
3. **Цикл**: повторять D5→R5 пока все 5 reviewers не скажут «no issues found»
4. **Лимит**: максимум 10 итераций, затем эскалация

---

## Sub-Agent Output Contract

Каждый sub-agent (debugger, reviewer, worker, etc.) ОБЯЗАН завершать работу структурированным ответом:

```
SUMMARY:    один абзац — что сделано и что произошло
CHANGES:    изменённые файлы с однострочным описанием; "None." если read-only
EVIDENCE:   path:line-range цитаты и ключевые находки; по одному bullet каждое
RISKS:      что может пойти не так / что родитель должен перепроверить
BLOCKERS:   что остановило; "None." если работа завершена чисто
```

Главный агент (main) использует эти поля для автоматического анализа результатов.

## Code Conventions

- **Импорты**: `src.` префикс для внутренних, стандартная библиотека → сторонние → `src`
- **Нейминг**: `snake_case` для функций/переменных, `PascalCase` для классов, `UPPER_CASE` для констант
- **Docstrings**: Google-style на всех публичных функциях
- **Комментарии**: На русском, описывают ЗАЧЕМ а не ЧТО
- **Запрещено**: TODO, FIXME, HACK — либо исправь сейчас, либо не пиши
